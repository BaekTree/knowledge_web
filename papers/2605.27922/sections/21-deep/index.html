Field

This section presents the concrete task designs and oracle evaluation methodology that operationalize the Harness-Bench framework across three representative agent workflows. Rather than defining abstract metrics, it demonstrates how realistic constraints and measurable outputs create a standardized evaluation surface. Each task card specifies an input workspace, a clearly defined agent objective, the concrete artifacts the agent must produce, and hard constraints that cannot be violated. This structure reflects the core insight that agent capability cannot be separated from the context in which it operates—a working solution to a database migration is not just correct code, but code that respects schema boundaries, runs idempotently under transaction isolation, and degrades safely if re-applied.

The oracle grading methodology moves beyond binary correctness to combine multiple evaluation dimensions. For code repair (fixing buggy implementations), the oracle verifies not only that bugs are fixed but also the quality of fixes—measured through completion rate, round efficiency (how many attempts the agent required), and adherence to constraints like minimal changes and preservation of intended behavior. This layered grading reflects a pragmatic reality: an agent that discovers the correct fix in one attempt is more valuable in production than one requiring ten iterations, and an agent that adds extraneous changes or "preemptive" fixes for hidden issues is less trustworthy than one that solves exactly the stated problem. The requirement for a # FIX: ... comment on each repair also captures a design principle often overlooked in benchmarks—that agent outputs must be human-reviewable and maintainable.

The database migration task exemplifies how constraints can encode real-world requirements into measurable criteria. The agent must repair an unsafe migration while satisfying six independent dimensions: data preservation (no user or dependent-order loss), deterministic cleanup of dirty email rows, email constraint enforcement, idempotency (safe to re-run), rollback capability, and postcheck coverage. The prohibition on editing schema and policy files creates an ownership boundary—the agent works within given constraints rather than redefining the problem. The requirement that the migration execute offline inside an explicit transaction and not corrupt data on a second run captures the actual operational constraints of production database work, where concurrency, atomicity, and repeatability are non-negotiable.

The support routing task extends the framework into professional workflows where correctness is not purely technical but policy-grounded. The agent must route tickets, cite the governing policy clause for each decision, assign priority levels, and prepare both customer-facing replies and escalation notes for human review. This task design recognizes that many agent workflows operate at the human-AI boundary rather than as fully autonomous systems. By requiring policy citations and escalation notes, the oracle grading can verify not just that the agent made a decision but that it reasoned transparently about the basis for that decision. The three-hour timeout (versus twelve minutes for code repair and twenty minutes for migrations) acknowledges that complex reasoning over policy text and order history is more computationally demanding than isolated code or schema tasks.

Together, these task cards establish a methodological baseline: the Harness-Bench evaluation combines workspace isolation (independent inputs and outputs), hard constraints that reflect operational reality, and multi-dimensional oracle grading that measures not just correctness but quality, efficiency, and trustworthiness. This design ensures that improvements measured in the benchmark translate to improvements in realistic agent deployment.
