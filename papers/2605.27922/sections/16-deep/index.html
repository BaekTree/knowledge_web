Introduction

The paper introduces the concept of a "harness" as a critical but understudied component of deployed agent systems. A harness is the engineering layer between a language model's outputs and their execution in external environments. It encompasses the mechanisms that organize context windows, manage tool availability, track system state, enforce permission boundaries, define constraints on model action, and implement recovery procedures when tasks fail. The motivation is pragmatic: while benchmarks traditionally evaluate model capability in isolation, real-world agent systems depend equally on how well the harness translates that capability into reliable, safe, and efficient action. The authors argue that harnesses shape multiple dimensions of practical performance—completion rates, computational cost, safety, robustness, and auditability—yet remain largely invisible in existing evaluation frameworks.

The existing agent evaluation landscape, despite rapid progress, leaves harness effects unmeasured. Static benchmarks like MMLU evaluate raw language understanding. Agent benchmarks like SWE-bench and WebArena test end-to-end task completion in executable environments. However, each of these approaches either abstracts away the harness (treating execution as a black box), conflates the harness with the entire agent system (making it impossible to isolate harness contributions), or fixes the harness across all tested models (preventing any measurement of how harness choices interact with model selection). As a result, practitioners lack a diagnostic protocol to understand whether performance differences come from the model, the harness configuration, or their interaction in a specific workflow.

Harness-Bench's core design insight is to invert the evaluation frame by making the harness a primary axis of comparison while controlling external task conditions. Rather than asking "which model is best," it asks "how do different model–harness pairings perform on the same tasks?" The benchmark fixes the task environment, execution budget (token limits or time), timeout policies, and evaluation criteria, but deliberately preserves the native execution behavior of each harness being tested. This design choice enables configuration-level diagnostics: measurements reflect real performance of actual harness implementations, not an idealized abstraction, yet isolate harness effects by holding tasks and evaluators constant across runs. The resulting data points toward whether harness differences matter independently of model choice.

The benchmark itself consists of 106 realistic, end-to-end agent tasks constructed from practical workflows users actually perform. Rather than synthetic microbenchmarks testing isolated capabilities, each task is a complete workflow—potentially spanning multiple steps, tool calls, state modifications, and decision points—executed in its own sandboxed, offline environment with task-specific configuration. This design preserves realism and reproducibility: tasks run offline to avoid benchmark drift from live services, yet span diverse operational domains including software engineering, data analysis, knowledge-work, and permission-sensitive stateful processes. Manual review for difficulty, solvability, and evaluation reliability ensures the benchmark exposes meaningful performance differences rather than noise.

These three contributions—the benchmark asset, the evaluation protocol, and the diagnostic analysis across 5,194 execution trajectories—form a foundation for understanding agent systems at the configuration level. Rather than attributing performance to a model's intrinsic capability, Harness-Bench enables practitioners to measure how specific architectural choices in the harness layer affect whether a given model can reliably complete realistic workflows. This perspective reflects a shift in agent evaluation: from comparing standalone models to characterizing integrated system configurations.
