05 5 Factor

Harness-Bench uses a factorial experimental design that systematically isolates the effects of different harness implementations from confounding variables in the task environment. The core strategy is to hold external task conditions constant—including the task prompt, initial sandbox state, computational budget, timeout limits, and evaluator setup—while varying two critical dimensions: the model backend and the harness configuration itself. This separation is essential because harness effects cannot be measured if the task environment, resource constraints, or evaluation criteria shift between experiments. By fixing these boundary conditions, the benchmark creates a controlled baseline where observed performance differences can be attributed to genuine differences in how each harness structures agent behavior and tool interaction.

The factors within each harness are further partitioned into two classes: those preserved as native behavior and those standardized. Rather than forcing each harness into an artificial uniform interface, Harness-Bench respects each harness's native execution model for critical details like prompting strategy, action format, tool interface design, and retry/recovery behavior. This preservation is intentional; it reflects how practitioners actually use these harnesses in the wild and prevents the benchmark from optimizing for a synthetic middle-ground that no real deployment would adopt. Permissions and available tools are held to a minimal required set, preventing any single harness from gaining unfair advantage through expanded access. This design decision acknowledges that standardization and authenticity are in tension, and resolves it by standardizing only where it matters for fair comparison, while keeping each harness's native strengths and weaknesses visible.

Table 2 presents results aggregated across both the 8 model backends and 106 tasks, revealing substantial performance variance across harnesses. NanoBot achieves the highest overall score of 76.2%, with NanoBot and Moltis showing strong completion rates above 78% and robust security properties (all harnesses reach 100%). The process efficiency metrics—tool use, consistency, and robustness—show that medium-performing harnesses like Hermes and ZeroClaw often exceed simpler baselines, suggesting that harness complexity does not automatically degrade reliability. Notably, token consumption varies by more than a factor of two: OpenClaw uses only 82.1K tokens across its tasks, while ZeroClaw and NullClaw consume 175.1K and 133.2K respectively, indicating that different harnesses make fundamentally different decisions about action granularity, retry strategies, and context management.

The reported separation of Codex as a model-bound agent is methodologically important: unlike configurable harnesses, Codex represents a monolithic system where model architecture and harness behavior are inseparable. By reporting its 80.4% score independently rather than averaging it into the multi-harness results, the benchmark acknowledges that comparing model-bound agents to configurable harnesses requires different statistical treatment. This methodological choice highlights the paper's contribution—Harness-Bench is designed to measure how harness design choices, independent of model selection, shape agent reliability and efficiency in realistic workflows where both factors coexist but interact in measurable ways.
