Self-Harness: Harnesses That Improve Themselves
arxiv:2606.09498

1 Introduction

This introduction frames an LLM agent as more than just a base model: its behavior is strongly shaped by the “harness,” meaning the surrounding system of prompts, tools, runtime rules, verification, and recovery logic. The core technical claim is that the same model can perform very differently under different harnesses, so the bottleneck is not only model capability but also how the model is embedded in the agent loop. The paper’s proposal, Self-Harness, is to let the agent improve its own harness instead of relying on human engineers or a stronger external agent. The design is an iterative improvement loop with three stages. First, Weakness Mining runs the current harness on tasks and clusters failed traces so the system can identify recurring failure modes rather than react to isolated errors. Second, Harness Proposal asks the agent to produce small, diverse edits that are explicitly tied to those failure patterns; this “minimal but targeted” constraint is important because it reduces the chance of broad, noisy prompt changes that accidentally fix one issue while harming others. Third, Proposal Validation uses regression tests and held-out tasks to accept only edits that improve performance without measurable degradation, which is the mechanism that guards against overfitting to observed failures. The experimental results in this section serve as motivation, not the full argument: they show that self-improvement can yield model-specific harnesses that generalize across tasks, and that the changes are often concrete operational fixes such as better file handling, dependency checking, or loop avoidance. This sets up the next part of the paper, which must explain how Self-Harness is implemented in practice and how those proposed edits are generated, evaluated, and merged into a working system.
