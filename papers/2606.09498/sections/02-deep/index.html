Self-Harness: Harnesses That Improve Themselves
arxiv:2606.09498

13 BASELINE SYSTEM PROMPT build system prompt

This section defines the starting point for Self-Harness: a deliberately minimal harness built on the DeepAgent SDK. The baseline only includes the default filesystem and shell tools plus a short system prompt, while the rest of the behavior is exposed through explicit configuration functions such as `build_bootstrap_instruction`, `build_execution_instruction`, `build_verification_instruction`, and `build_failure_recovery_instruction`. The key technical idea is that the harness itself becomes the object of optimization. Instead of tuning the model weights, Self-Harness edits the surrounding execution policy that shapes how the model plans, uses tools, recovers from errors, and decides when to verify results. The design choice to keep the harness small is important. It creates a clean causal setup: any performance change can be attributed to harness edits rather than to a richer toolset, a different budget, or a more permissive environment. The paper also fixes a held-in/held-out split before proposing edits. The held-in split supplies traces and failure evidence to the proposer, while the held-out split remains hidden and serves as the promotion gate. That separation is what makes the improvements meaningful: the system is not just memorizing specific failures, but learning reusable execution mechanisms that generalize across tasks. This section sets up the empirical part of the paper. Once the baseline harness and editable interfaces are defined, the next step is to let Self-Harness iteratively propose changes, test them under the promotion rule, and measure whether the promoted harness improves pass rate on Terminal-Bench-2.0. The later results and trace analyses rely on this setup to show that gains come from targeted changes such as earlier artifact creation, better tool-error recovery, and stronger verification discipline, rather than from arbitrary prompt tweaks.
