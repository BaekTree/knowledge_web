Self-Harness: Harnesses That Improve Themselves
arxiv:2606.09498

Section 6 deeper explanation

Harness

The section introduces the paper’s core object: a **harness**, meaning the control layer that wraps an agent and determines how it is instantiated, prompted, equipped with tools, and verified. The initial harness is built on the DeepAgent SDK, but the authors deliberately keep it minimal: a short system prompt aimed at the benchmark, plus the default filesystem and shell tools. That choice matters because it gives the system a simple, stable starting point, so any later performance changes are easier to attribute to harness edits rather than to a complex initial setup. A key constraint is that Self-Harness can only modify the **harness definition file** that configures DeepAgent. In other words, the system is not allowed to rewrite the whole agent arbitrarily; it can only adjust declared configuration points such as instructions, tool access, and verification guidance. This is a controlled design space. By limiting edits to these surfaces, the method keeps the search over candidate harnesses structured and interpretable, while still allowing meaningful improvements in how the agent behaves. A candidate harness at time `t`, written as `h_t^(j)`, is one such proposed revision. This section sets up the next step of the paper: a self-improvement loop over harness candidates. Once the baseline harness and its editable parameters are defined, the rest of the method can ask the system to propose, evaluate, and select better harnesses. The minimal initial setup is therefore not just an implementation detail; it is the reference point from which the paper measures whether self-modification actually helps.
