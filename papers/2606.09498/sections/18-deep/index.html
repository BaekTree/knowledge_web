Section 18 deeper explanation

Trace-Level Analysis of Accepted Edits

This section is a trace-level sanity check on what the accepted harness edits actually do at runtime. Instead of treating the harness as a static prompt, the paper compares before-and-after execution traces and asks whether the edited harness changes the agent’s workflow in the same way the earlier diagnostics predicted. That makes the claim stronger than a raw score improvement: it shows that the improvements come from behavior-level interventions, not just from a larger or more verbose prompt. In the paper’s logic, these edits are the retained outputs of the earlier Weakness Mining and Proposal Validation stages, so this section is where the authors verify that the selected edits really address the failures they were chosen for.

The MiniMax M2.5 example shows a shift from vague search behavior to bounded artifact production. The accepted edit changes the bootstrap instruction so the agent does not merely look for the smallest edit surface; it must identify the required output artifact and start producing it early. The added runtime limit on total tool messages is important because it constrains unproductive exploration and forces a decision point. In the trace, that manifests as a move from prolonged dataset inspection to a concrete sequence: locate the relevant split, compute the needed count, write the answer file, and read it back. The design choice here is not just “be more concise,” but “optimize the control flow around the success condition the verifier will check.”

Qwen3.5 receives a different fix because its dominant failure mode is different. Its edited harness adds rules for dependency prechecking, loop breaking, retry discipline, and recovery after tool errors. These are not generic safety phrases; they are procedural constraints that reshape how the agent reacts when its first attempt fails. In the failure trace, the agent repeatedly edits the same extractor, hits overwrite errors, and ends by deleting the file the verifier expects. After the harness edit, a tool-error-triggered prompt redirects the agent back to the missing artifact, so it recreates the extractor, repairs the parsing, writes the output file, and validates the JSON result before stopping. This is a targeted recovery policy, not a blanket prompt expansion.

GLM-5 is optimized along yet another axis: environment persistence and the transition from exploration to implementation. The accepted edits explicitly remind the agent that installed tools and path changes must survive across shell sessions, and they add a rule to switch from exploration to implementation once the agent has spent too long without producing required artifacts. In the trace, that changes a download-heavy, timeout-prone workflow into one that checks sources earlier, repairs a failing render check, and only then finalizes. Across the three cases, the section reinforces the paper’s broader argument from the Main Results: Self-Harness works because it induces model-specific workflow corrections, with retained edits carrying forward the most useful behavioral fixes discovered during the earlier stages.
