Trace-Level Analysis of Accepted Edits

This section uses trace-level analysis to show how accepted harness edits change behavior, not just whether they improve scores. A trace is the step-by-step record of an agent interacting with tools, so comparing before-and-after traces lets the authors infer which failure mode the edit is targeting. The key idea is that Self-Harness does not apply one universal prompt tweak; instead, it promotes model-specific changes that address the dominant errors seen in each model’s initial harness. In other words, the edits are diagnostic: they are chosen because the trace suggests a particular breakdown, such as endless exploration, repeated tool misuse, or failure to preserve environment state.

The MiniMax M2.5 example illustrates a design choice centered on early artifact creation and bounded execution. The edited instruction shifts the agent from vaguely “finding the smallest edit surface” to explicitly identifying the required output artifact and creating it early. That matters because many agent failures come from over-exploration: the model keeps searching and never commits to a concrete deliverable. Adding a cap on total tool messages further encourages termination and redirection. The trace changes accordingly: instead of wandering through the dataset, the agent identifies the relevant split, computes the count, writes the answer file, and checks it. The workflow is designed to force progress toward the verifier’s actual requirement, not just more reasoning.

For Qwen3.5, the promoted harness adds protections against repeated failure loops and improves recovery after tool errors. The added constraints for dependency prechecking and retry discipline are meant to prevent the agent from repeatedly hammering the same broken command or artifact. The important intuition is that some failures are not due to lack of knowledge but due to poor control flow under error conditions. After the edit, a tool error triggers a more useful system prompt: the agent returns to the missing artifact, recreates the extractor, fixes parsing, writes the output file, and validates the JSON. The point is not simply “try harder,” but “recover in a way that preserves the required deliverable.”

The GLM-5 edits focus on environment persistence and on switching from exploration to implementation once evidence shows that exploration is no longer productive. This is crucial in shell-based tasks, where changes can disappear across sessions if they are not made persistent. The edited harness also tells the agent to verify tool accessibility after modifying the environment, which is a practical guard against silent setup failures. These trace changes set up the paper’s next step: connecting qualitative behavior to the quantitative gains. The section argues that the improved scores in Figure~ are explained by targeted workflow shifts, so the next analysis can treat those edited harnesses as causal mechanisms rather than just better prompts.
