Harness

This section introduces the “harness” as the control layer around the agent, not the agent itself. The initial harness is built on the DeepAgent SDK, but it is intentionally stripped down: a short system prompt aimed at the benchmark, plus the standard filesystem and shell tools. That design keeps the starting point simple and well-defined, so any later performance changes can be attributed to harness modifications rather than to a complicated baseline implementation. A key constraint is that Self-Harness can only edit the harness definition file, which is the configuration that tells DeepAgent how to be instantiated and managed. In practice, that means the system is not free to rewrite arbitrary code; it can only modify declared control surfaces such as the instructions, tool set, and verification guidance. This is a deliberate modularity choice: by limiting edits to explicit configuration points, the method makes harness changes safer, easier to compare, and easier to interpret as candidate variants \(h_t^{(j)}\). The point of this setup is to create a controlled search space for self-improvement. Each candidate harness is a different configuration of the same underlying agent framework, so the paper can study whether better prompting, tool access, or verification rules improve benchmark performance. This section therefore sets up the next step in the paper: generating, evaluating, and selecting among harness candidates as part of the self-improvement loop.
