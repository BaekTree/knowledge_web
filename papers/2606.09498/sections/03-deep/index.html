Self-Harness: Harnesses That Improve Themselves
arXiv:2606.09498

Section 3 deeper explanation

From prompts to agent harnesses

This section broadens the idea of “prompting” into a larger control system. In classic prompt engineering, the main lever is text: instructions, examples, retrieved facts, or memory injected into the model’s context. Context engineering extends that by treating the input as something dynamically assembled from multiple sources. The paper then argues that agentic systems go one step further: the model is not just producing one answer, but operating inside an environment where it can act, observe results, use tools, receive feedback, and be constrained by runtime policies. In that setting, behavior is shaped as much by the surrounding machinery as by the model itself. The paper calls that surrounding machinery the “harness.” A harness includes prompts, tools, memory, verification checks, permission rules, adapters, and execution logic that mediate between the model and the world. This definition is important because many failures in agents are not pure reasoning failures. An agent can claim success without checking whether a file was actually written, repeat a bad action because there is no recovery path, or lose track of the authoritative source in a long workflow. Those are system-level failures: the model may be capable, but the harness lets the wrong behavior persist. The design choice here is to treat the harness as a first-class object rather than a fixed wrapper around the model. That framing explains why improving agent performance often requires changing policies, feedback loops, and tool orchestration, not just rewriting the prompt. It also sets up the next step of the paper: if the harness strongly shapes behavior, then the harness itself can be studied, evaluated, and improved. That is the bridge to “self-harness,” where the system learns to redesign its own surrounding control layer.
