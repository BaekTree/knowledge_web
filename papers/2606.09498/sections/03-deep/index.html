From prompts to agent harnesses

Prompt engineering and context engineering have demonstrated that fixed language models can be effectively steered through careful instruction design, demonstrations, retrieved evidence, memory structures, and tool availability. However, agentic systems represent a fundamental shift in this control paradigm: rather than steering a model through a single input, they expand the control surface to encompass an entire execution environment. In this broader setting, a model acts, observes the consequences of its actions, uses available tools, receives feedback from the environment, and follows explicit runtime policies. This shift from input-time control to runtime control—illustrated by systems like ReAct, SWE-agent, Claude Code, and OpenClaw—reveals that long-horizon agent behavior depends on far more than the initial prompt.

This expanded control surface is what the paper terms a "harness": the surrounding system layer comprising prompts, tools, memory mechanisms, verification rules, permission policies, adapters, and runtime control logic that mediates between the model and its environment. The key distinction from classical prompt engineering is that a harness is not simply a well-crafted instruction. Instead, it is a structured composition of interacting components, each governing different aspects of agent behavior. Tools define what actions are available and how feedback is returned. Memory mechanisms determine what information the model can access and in what form. Permission policies constrain which actions are actually executable. Verification rules decide when to accept or reject the model's outputs. Runtime control decides whether to retry, escalate, or abort. These components interact: a tool without proper permission policies may allow unsafe actions; a memory system without verification may propagate stale information; runtime control without clear termination conditions may enter infinite loops.

Agent failures frequently emerge not from isolated deficiencies in the model's reasoning but from failures in these systemic interactions. The paper identifies several characteristic failure modes: an agent may report success without actually verifying an artifact, suggesting that observation loops are incomplete; it may retry an unproductive action pattern, indicating that feedback mechanisms lack sufficient signal to escape local minima; it may lose the source of truth in a long context, pointing to memory system degradation; or it may lack a recovery action when encountering an unexpected state, showing that runtime policies are incomplete. Critically, these failures cannot be fixed by revising the prompt alone. If an artifact verification rule is missing from the harness, no prompt will reliably make the model self-verify. If runtime control does not allow backtracking, no instruction will create that capability.

The insight that agent behavior emerges from harness-level interactions—not from the model's intrinsic capabilities—motivates a different approach to improvement. Rather than iterating on prompt text, we must treat the harness as a structured object suitable for systematic modification and evaluation. This structured representation becomes essential when seeking to improve harnesses automatically: we must be able to articulate what can vary (specific tools, memory retention policies, verification thresholds, control flow patterns) and measure how those variations affect behavior. Without this structure, improvement requires manual experimentation on an undecomposed system. With it, we can apply techniques like those explored later in the paper to explore diverse harness configurations and identify those most effective for a given task.
