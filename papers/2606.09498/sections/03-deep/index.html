Section 3 deeper explanation

From prompts to agent harnesses

This section broadens the scope from “prompting a model” to “building the whole system around the model.” Prompt engineering and context engineering showed that a fixed model can be steered by the inputs it receives: instructions, few-shot examples, retrieved documents, memory, and tool state. The key step here is that agentic systems make this control surface operational. The model is no longer just answering once; it is acting in an environment, observing consequences, using tools, receiving feedback, and following runtime policies. In that sense, the model is only one component inside a larger loop.

The paper uses the word “harness” for that surrounding system layer: prompts, tools, memory, verification rules, permission policies, adapters, and runtime mechanisms that mediate between the model and the environment. This is a deliberate design choice because many real failures happen outside the model’s raw language ability. An agent can claim it succeeded without checking the artifact, keep repeating a bad action, forget the authoritative source in a long context, or fail to recover after an error. Those are not just “bad answers”; they are failures of orchestration. The harness concept groups these pieces together so the authors can reason about them as a single engineering object instead of as unrelated prompt tweaks.

The intuition is that behavior emerges from the interaction of instructions, observations, tools, and control logic. A good harness does not merely ask the model to “be careful”; it constrains what actions are possible, what evidence must be checked, when a retry is allowed, and how success is verified. That is why the section emphasizes runtime structure rather than text alone. Systems like ReAct, SWE-agent, Claude Code, and SemaClaw/OpenClaw are cited as examples because they show that surrounding mechanisms can materially change long-horizon performance, especially in software tasks where action sequences and verification matter more than single-turn fluency.

This framing sets up the next step in the paper: if the harness is where many failures originate, then improving the agent means improving the harness itself, not just the base model or the prompt template. The section therefore provides the conceptual bridge to self-harnessing: a system that can inspect, adapt, and improve the policies and mechanisms that govern its own operation.
