Metrics

The section defines the paper’s main evaluation metric: **Pass (%)**, the share of task attempts that satisfy the benchmark’s verifier. In practice, each harness candidate is tested on two repeated attempts per task, and the pass rate is computed from those outcomes unless the paper says otherwise. The important idea is that the metric is not measuring how often the harness produces a nice-looking intermediate trace; it is measuring whether the run ends in a verifiably correct final result. That makes the metric objective and comparable across harnesses. This design choice matters because harnesses can vary in how they guide the model, but the benchmark needs a single, stable success signal. By using a verifier rather than subjective judging, the authors avoid ambiguity about partial progress or stylistic quality. The repeated attempts help reduce noise from stochastic model behavior: if one run succeeds and another fails, the Pass rate captures the harness’s average reliability under the same configuration and evaluation protocol. For Terminal-Bench-2.0 specifically, the success signal comes from checking the final container state, which matches the benchmark’s focus on end-to-end task completion rather than intermediate steps. This metric sets up the next stage of the paper because self-harnessing only makes sense if improvements can be measured consistently. Once Pass (%) is fixed as the yardstick, the authors can compare candidate harnesses, track whether self-modifications help or hurt, and run iterative improvement loops with a clear optimization target. In other words, the metric becomes the feedback signal that drives the self-improvement process described later.
