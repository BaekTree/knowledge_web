Benchmarks

This section explains why Terminal-Bench-2.0 is a strong evaluation target for Self-Harness. The benchmark is agentic rather than static: the model must operate through a terminal, take multiple steps, and adapt to what happens in the environment. That matters for a system like Self-Harness because the paper is studying whether the harness itself can be improved, not just whether a fixed prompt answers questions well. A benchmark with realistic execution traces and deterministic verification gives a clean way to observe whether harness edits improve the agent’s ability to execute tools, manage state, and finish tasks reliably.

The key design choice is to evaluate on a fixed 64-case subset rather than the full 89-task benchmark. The paper excludes cases that depend on unstable external web resources and cases that require multimodal inputs. This is not just a convenience filter. It removes sources of variance that would obscure the effect of the harness: if a task can fail because a website changed or a resource disappeared, then the outcome is partly outside the agent’s control. Likewise, multimodal tasks would demand input handling that the initial minimal harness does not even support, so those tasks would measure missing infrastructure rather than harness improvement. In other words, the subset is chosen to make the benchmark more diagnostic of harness quality.

Terminal-Bench-2.0 stresses the exact behaviors that harness evolution should improve. The tasks require artifact handling, meaning the agent must create, modify, and preserve files correctly; command use, meaning it must choose and sequence shell operations effectively; verification behavior, meaning it must check whether its outputs satisfy the task; and recovery from errors, meaning it must respond intelligently when commands fail or intermediate state is wrong. These are the same kinds of operational skills that a better harness should support, so performance changes can be meaningfully attributed to the harness rather than to superficial prompt differences.

This section also connects directly to the rest of the paper’s evaluation protocol. The benchmark subset and its filtering criteria are part of the “Splits and protocol” story: they define what is in scope for comparison and what is intentionally excluded. The resulting scores then feed into the Metrics and Main Results sections, where the paper can show whether self-edits to the harness improve success rates in a setting with deterministic judging. The broader point is that Self-Harness is evaluated in an environment where progress is both realistic and measurable.
