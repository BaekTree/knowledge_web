Weakness Mining Identifying Failure Patterns from Clustered

This section defines the paper’s first evidence-collection stage: turn raw execution failures into structured inputs for harness improvement. The setup is deliberately controlled. A fixed model M is run under the current harness h_t on a held-in split D_in, so the only changing component across rounds is the harness itself. For each instance x_i, the run produces the model output y_i and an execution trace τ_i, and the evaluator E converts that trace plus output into an outcome z_i such as pass or fail. The resulting record r_i = (x_i, τ_i, y_i, z_i) is the atomic unit of analysis. This matters because it creates a clean causal frame: if a failure pattern changes across rounds, the paper can attribute that change to the harness revision rather than to a different model or evaluator.

The main design choice is to focus only on failed records F_t, rather than all runs. The intuition is that successful cases are less informative for fixing the harness, while failures expose where the current harness is brittle. But the paper does not stop at “the run failed.” It asks why it failed in a verifier-grounded way. Each failure is summarized by a signature σ(r_i) = (c_i, q_i, m_i), where c_i is the terminal cause identified by the verifier, q_i captures how the relevant agent behavior contributed causally, and m_i abstracts the reusable mechanism behind that behavior. This decomposition avoids a common mistake: grouping failures merely because they end with the same surface symptom. A timeout, for example, may arise from different internal behaviors, and those would call for different harness edits.

Clustering is therefore deterministic and exact over the failure signature. Two failed traces are grouped together only if they agree on the verifier’s final rejection reason, the causal role of the behavior, and the underlying mechanism. That is a strong constraint, but it is the point: the system is not trying to discover fuzzy semantic similarity among traces. It is trying to aggregate cases that plausibly admit the same harness-level intervention. In other words, the cluster is an evidence object, not a latent topic model. For each cluster C_k, the system builds a structured pattern summary with support size, representative instances, shared symptoms, verifier evidence, and inferred mechanism.

The final step is to rank these clusters into an evidence bundle B_t. Ordering by support and estimated actionability ensures that downstream proposal generation sees the most recurring and promising weaknesses first. This is what connects the section to the rest of the paper: the bundle feeds the proposal stage, where candidate harness modifications are explored, and later the validation stage, where those candidates are tested for robust improvement. Compared with the earlier “Preliminary” framing, this section operationalizes the core self-improvement loop: diagnose failures at the trace level, package them as reusable evidence, and hand them off in a form that supports minimal, targeted harness edits.
