Self-Harness An Iterative Loop for Model-Specific Harness Im

This section introduces Self-Harness as an iterative self-improvement loop for agent “harnesses,” meaning the surrounding control code that steers a model’s behavior during execution. The key idea is to sit between two extremes: on one side, human harness engineering, where experts manually inspect failures and rewrite the harness; on the other, external optimizer methods, where harness design is treated as a search problem. Self-Harness instead keeps the model fixed and lets it improve the harness around itself, using execution evidence rather than abstract guesses about what might work. The loop has three main technical stages. First, the current harness is run and its execution traces are clustered to find recurring failure patterns. Clustering matters because single failures can be noisy, but repeated trace structures usually indicate a real mechanism, such as a bad tool-selection branch or an overly rigid stopping rule. Second, the same model is reused in a proposer role, but now it is given structured evidence and asked to produce small, diverse edits. The “minimal” requirement is deliberate: the system is not trying to redesign the whole agent, only to patch the specific weakness while preserving the broader control architecture. Third, each candidate edit is regression-tested on held-out tasks, and only improvements that clear an acceptance threshold are kept. That acceptance rule is what prevents the loop from overfitting to a narrow set of observed failures. The design reflects a practical tradeoff: use the model’s own generative ability to suggest fixes, but anchor those suggestions in measured behavior and test-time validation. This section sets up the next part of the paper by defining the actual optimization loop and its criteria for safe adoption. In other words, it moves from the intuition of “self-improving harnesses” to the mechanics of how candidate changes are proposed, checked, and promoted.
