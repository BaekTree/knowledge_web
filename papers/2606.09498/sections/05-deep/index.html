Self-Harness An Iterative Loop for Model-Specific Harness Im

Self-Harness proposes an iterative feedback loop that positions itself between two extremes: manual human engineering of harnesses, which relies on expert intuition but doesn't scale, and external optimizer approaches, which treat harness design as a black-box parameter search. The core insight is that a model can iteratively refine its own execution framework by analyzing evidence from its actual behavior. Rather than searching over arbitrary design choices or relying on human judgment, Self-Harness grounds improvement in concrete failure patterns extracted from execution traces. This positions the model as both the beneficiary and the architect of its harness, creating a closed-loop system where each iteration produces actionable insights that drive the next round of modifications.

The architectural flow of the feedback loop consists of four distinct stages that together form a cycle. First, the evaluation system runs the current harness against a task set and clusters the resulting execution traces to identify recurring failure patterns—the structured evidence mentioned in the Weakness Mining section. Second, the same model is invoked in a proposer role (a different prompt or context) and tasked with generating candidate harness modifications based on this evidence. These modifications are deliberately designed to be diverse yet minimal, targeting specific failure mechanisms without overhauling the entire control architecture. Third, candidate edits are validated through regression testing on held-out tasks to ensure they don't degrade performance on cases where the harness already works. Finally, an explicit acceptance rule determines which edits are promoted—only those that demonstrably improve performance without introducing unacceptable regressions. This stage-by-stage design ensures that improvements are validated before commitment rather than applied speculatively.

The model-specificity of Self-Harness is a critical departure from universal optimizer frameworks. Rather than searching for harness patterns that work across all models, Self-Harness acknowledges that different models have different failure modes, reasoning patterns, and optimal architectural choices. The same model that profits from the harness improvement is the same model proposing the fix, which creates a form of epistemic alignment—the model understands its own failure patterns more intimately than a generic optimizer can. This approach recognizes that a 70B parameter model and a 7B parameter model might require fundamentally different harness structures, different instruction formats, or different planning depths. The iterative cycle is therefore model-specific by design: you run the loop for model A to convergence, and that optimized harness may be entirely different from the harness optimized for model B.

Convergence and stopping criteria are implicit in the validation stage. The loop terminates either when successive iterations produce no edits that pass the acceptance rule (indicating that remaining failures are harder to address through architectural changes), or when the acceptance rule filters out all proposed edits, signaling that further modifications risk introducing regressions. In practice, the number of iterations required depends on model capacity and task complexity, but the explicit regression-testing validation ensures that the system cannot drift into deteriorating performance. This differs from human engineering, where the stopping point is often arbitrary or expert-dependent, and from external optimizers, where computational budget is the limiting factor. Here, stopping is driven by the evidence itself: when the model exhausts the low-hanging fruit and proposed improvements no longer clear the regression bar, iteration ceases naturally.

The Self-Harness approach ultimately bridges individual model agency and systematic improvement. By making the model the primary architect of its own harness while enforcing rigorous validation, the method combines the interpretability and agency of human engineering with the scalability and objectivity of automated optimization. This iterative structure connects closely to the Proposal Validation section, which details how regression testing enforces robustness, ensuring that each iteration genuinely strengthens the overall system rather than merely optimizing for a narrow set of visible failures.
