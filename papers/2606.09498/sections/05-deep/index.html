Self-Harness: Harnesses That Improve Themselves
arxiv:2606.09498

Section 5 deeper explanation

From prompts to agent harnesses

This section broadens the usual notion of “prompt engineering” into a larger systems view. A fixed model can be influenced not only by the text prompt, but also by demonstrations, retrieved documents, memory, tool outputs, and other runtime inputs. The paper then extends that idea to agentic systems, where the model is no longer just answering once: it acts in an environment, observes results, calls tools, receives feedback, and follows policies during execution. In that setting, the important object is the surrounding control system, not the model alone. The authors call this surrounding layer the “harness.” A harness includes prompts, tools, memory, verification rules, permission checks, adapters, and any runtime mechanism that mediates between the model and the world. The design choice here is to treat these pieces as one technical unit because many failures arise from their interaction: an agent may claim success without validating an output, repeat a bad action loop, forget the authoritative source in a long session, or get stuck because it has no recovery path. In other words, the failure is often in orchestration, state management, or policy enforcement, not in the base model’s raw capability. That is why improving agent performance often requires changing the harness itself, not just rewriting instructions. This framing sets up the paper’s next step: if the harness is a first-class system component, then it can also be improved systematically. The section prepares the reader for self-harnessing by establishing that agent behavior is shaped by an editable execution environment. Once that is accepted, the natural question becomes how to measure, modify, and iteratively improve the harness so the whole agent loop becomes more reliable over long-horizon tasks.
