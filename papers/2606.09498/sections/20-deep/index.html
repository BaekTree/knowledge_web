Model inference services

The paper's experimental validation requires testing the self-harness framework across diverse model architectures and inference configurations to demonstrate its generalizability. This section describes a hybrid infrastructure strategy: leveraging hosted inference services for some models while maintaining local deployment for others. This dual approach reflects a fundamental trade-off between operational simplicity and experimental control. MiniMax M2.5 and GLM-5 were accessed through their respective hosted platforms—MiniMax's proprietary API and OpenRouter's aggregated endpoint for GLM-5—allowing the researchers to evaluate harness improvement without managing these inference backends directly. The local deployment of Qwen3.5-35B-A3B on four NVIDIA H200 GPUs using SGLang represents the complementary strategy, enabling fine-grained control over inference parameters and resource allocation during harness experiments.

The choice of hosted APIs introduces specific constraints and capabilities that shape how harness experiments proceed. Hosted services provide abstraction from underlying infrastructure complexity, reducing operational overhead and allowing the research team to focus on the self-harness algorithm itself rather than model serving infrastructure. However, this benefit comes with less direct observability and control over inference dynamics—rate limits, token quotas, and API versioning become external constraints that the harness must operate within. The paper's specification of access dates (May 2026) and endpoint URLs is critical for reproducibility; hosted API endpoints can drift in performance characteristics, rate limits, or availability, making explicit version tracking essential for future reproduction attempts. The reliance on OpenRouter for GLM-5 access adds an additional abstraction layer, as the endpoint aggregator may route requests across different underlying infrastructure, potentially introducing variability in latency and behavior.

The local deployment strategy for Qwen addresses different reproducibility and control concerns. By containerizing the inference service using a specific SGLang Docker image version (v0.5.12-cu129), the researchers establish a fixed inference environment that can be rebuilt identically across different experimental runs. The H200 GPU configuration provides consistent computational resources and enables direct measurement of how the harness adapts to specific hardware characteristics. This local setup allows experiments to vary inference parameters—batch sizes, attention mechanisms, quantization settings—in ways that hosted APIs typically restrict. The choice of four H200s suggests experiments were designed around a specific throughput requirement or multi-GPU inference pattern, indicating the harness validation spans scenarios where model serving infrastructure itself becomes a variable the self-improvement mechanism must consider.

The heterogeneity of inference services—hosted APIs with external rate limits and local deployment with direct resource control—creates a realistic experimental landscape that tests whether self-harness improvements generalize across different operational constraints. The paper's broader thesis about harnesses improving themselves depends on validating this behavior across diverse conditions; a harness that only improves under a single inference configuration would have limited practical value. By measuring harness behavior against MiniMax and GLM-5 through their hosted endpoints and against Qwen through local infrastructure, the experiments implicitly test whether self-improvement mechanisms can discover and exploit infrastructure-specific optimization opportunities or whether improvements remain portable across different backends. This infrastructure diversity is therefore not incidental but essential to the paper's claim about harness generalizability.
