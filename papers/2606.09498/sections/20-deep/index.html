Model inference services

This section specifies the execution environment for each model, which is more important than it might first appear. MiniMax M2.5 and GLM-5 were not run on the authors’ own hardware; they were accessed through external hosted inference services, namely MiniMax’s API and OpenRouter. By contrast, Qwen3.5-35B-A3B was served locally on four NVIDIA H200 GPUs using SGLang inside an internal image based on the public lmsysorg/sglang Docker build. That split means the paper is comparing models under two different operational regimes: remote API inference for convenience and provider-managed serving, versus self-hosted inference for direct control over runtime, hardware, and software stack.

The key technical distinction is that hosted API inference abstracts away the server implementation. The user sends a prompt, the provider handles scheduling, batching, decoding, rate limits, and backend model updates, and the caller only sees the returned text. Local GPU deployment exposes those details to the experimenter: they control the Docker image, the inference engine, the GPU count, and presumably the decoding settings. This matters because the same model name can behave differently depending on the serving stack, tokenizer version, quantization, max context handling, or sampling defaults. In other words, “the model” in practice is model plus serving environment.

Those serving details are essential for reproducibility and attribution. Reproducibility requires knowing not just which checkpoint was used, but where it was obtained, what backend served it, and when that service was accessed. The cited URLs and access month anchor the exact API endpoints that were available during the experiments, which is especially important for hosted services that can change without warning. Attribution is also clearer: MiniMax and OpenRouter are credited as the inference providers for the two remotely served models, while the Qwen model’s results are tied to the authors’ own local deployment setup.

This section connects directly to the paper’s broader experimental methodology. The Models section tells you which systems were evaluated; the Terminal-Bench-20 configuration defines the task environment; and the Experiments section interprets outcomes. Here, the serving setup fills in the missing bridge between the abstract model list and the actual results. Without it, readers could not tell whether a performance difference came from the model itself or from differences in how inference was executed.
