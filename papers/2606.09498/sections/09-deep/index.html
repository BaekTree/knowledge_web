Proposal Validation Ensuring Robust Improvement through Regr

This section describes the gatekeeper that decides whether a proposed harness edit becomes part of the live harness. The core idea is that a proposal is never trusted just because it sounds plausible or fixes the failure cases that motivated it. Instead, it is turned into a candidate harness and run through the same evaluator used on the current harness. The evaluator is split into a held-in set, D_in, and a held-out set, D_ho. The held-in split checks whether the edit addresses the evidence that triggered the proposal, while the held-out split checks for regressions on cases the proposer did not directly observe. That split is central to the paper’s goal: self-improvement should not only repair visible weaknesses, but do so without silently breaking other behavior.

Technically, the paper measures performance with pass counts on each split. For a candidate harness h_t^(j), the improvements are defined relative to the current harness h_t as differences in the number of passed tasks on D_in and D_ho. This makes the comparison local and incremental: the system is not asking whether the new harness is absolutely good, but whether it is better than the current one on each evaluation slice. The acceptance rule is conservative. A proposal is accepted only if it improves at least one split and does not degrade the other. In effect, the candidate must move the performance vector in a non-worsening direction and make strict progress somewhere. That prevents “net positive” changes that hide a tradeoff, such as improving the held-in tasks while regressing on held-out behavior.

The reasoning behind this rule is straightforward: if a harness is allowed to optimize only the evidence that created the proposal, it will overfit to its own self-generated diagnosis loop. The held-out split acts like a regression test for the unknown unknowns that were not visible to the proposer. The same logic extends to stochastic evaluation. When task outcomes vary across runs, the system repeats evaluation and aggregates pass counts before applying the rule. This reduces the chance that a lucky sample causes an edit to be promoted. The result is a promotion policy that is intentionally cautious, preferring stability and reproducibility over aggressive acceptance.

This mechanism connects directly to the paper’s broader self-harnessing thesis. The paper is not just generating edits; it is building a disciplined improvement pipeline where each transition in harness state is validated, logged, and auditable. Compatible edits can be merged when they jointly satisfy the rule, while rejected proposals remain in the record without affecting the active harness. That makes the harness lineage inspectable and supports iterative refinement without losing provenance. In short, this section supplies the safety and quality-control layer that lets self-improvement happen repeatedly without collapsing into unbounded drift.
