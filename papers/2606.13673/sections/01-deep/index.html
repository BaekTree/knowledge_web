1 3D Reconstruction

The reconstruction step is the paper’s first move from raw perception into a geometric world model. The call recon = tools.Reconstruct.Reconstruct(InputImages) is doing more than loading an image: it asks a dedicated reconstruction primitive to convert one or more input views into a structured 3D representation that later reasoning steps can operate on. In practical terms, that representation is the agent’s spatial working memory, replacing ambiguous 2D pixels with an explicit scene proxy such as depth, point cloud, mesh, or another geometry-backed form.

The main design choice here is to separate geometry extraction from language reasoning. Instead of making the VLM infer 3D structure implicitly from the pixels every time it answers a question, SpatialClaw gives it a tool that performs reconstruction once and exposes the result as recon. That matters because 3D questions often depend on occlusion, scale, relative pose, and viewpoint changes, all of which are much easier to reason about after the scene has been normalized into a common geometric frame. The assignment syntax also reflects the paper’s broader interface philosophy: every step should produce reusable state, not just a one-off answer.

There is no heavy mathematical derivation in this snippet, but the intuition is still important. Reconstruction is an estimation problem: given multiple 2D observations, infer a coherent 3D structure that best explains them. If multiple views are available, the tool can fuse them to reduce ambiguity and fill in occluded regions; if the inputs are sparse, the tool still provides the best geometric hypothesis available. The output is intentionally abstracted behind a tool boundary, because the paper’s point is not a new reconstruction algorithm, but a better action interface for using reconstruction as part of a larger agent loop.

This section connects directly to the paper’s broader thesis that agentic spatial reasoning should be iterative and geometry-aware. SpatialClaw’s code-based interface lets the model generate this reconstruction step, inspect the result, and then decide what to do next based on the intermediate geometry. That is the key advantage over fixed tool-call pipelines or single-pass programs: reconstruction becomes an editable, inspectable state that downstream spatial operations can build on.
